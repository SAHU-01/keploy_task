<HeaderSection title="Mux + Postgres Quickstart" />

Explore how Keploy captures binary Postgres traffic and generates automated tests for a Product Catalog app.

<Callout type="info">
  **Why Keploy?** Tired of writing `httptest` boilerplate and wiring up `sqlmock` for every handler? Keploy intercepts real API + database traffic and turns it into test cases automatically. Zero test code, zero fake data.
</Callout>

<Prerequisites>
  Before diving into the integration, ensure your environment is set up for traffic interception:

  *   **Keploy Binary**: Installed on your local machine.
  *   **Docker & Docker Compose**: Installed and running.

  Don't have Keploy installed yet? <span className="inline-flex items-center gap-1.5 font-semibold text-orange-500 animate-pulse transition-all"><Info className="h-4 w-4" /> [Go to Installation Guide](https://keploy.io/docs/server/install/)</span>
</Prerequisites>

<Step number={1} title="Clone the Sample App">
  <CodeTrigger file="main.go">
    Clone the repository and install dependencies to get started with the Mux-SQL sample.

    <CodeGroup>
      <CodeBlock label="Terminal" filename="terminal">
      ```bash
      git clone https://github.com/keploy/samples-go.git
      cd samples-go/mux-sql
      go mod download
      ```
      </CodeBlock>
    </CodeGroup>
  </CodeTrigger>
</Step>

<Step number={2} title="Configure for Docker">
  <CodeTrigger file="main.go" lines={[17, 22]}>
    Open `main.go`. Update the database host from `localhost` to `postgres` so the app can find the database container inside the Docker network.
  </CodeTrigger>

  <CodeGroup>
    <CodeBlock label="main.go" filename="main.go">
    ```go
    // Inside func main() initialization:
-   "localhost", // postgres host
+   "postgres", // Change localhost to postgres when using Docker
    ```
    </CodeBlock>
  </CodeGroup>
</Step>

<Step number={3} title="Lights, Camera, Record!">
  <CodeTrigger file="test-1.yml">
    ### Infrastructure Readiness
    Once your containers are healthy, you should see the services networked correctly in your terminal.

    ![Service Orchestration](/assets/quickstart/go/docker/mux-sql/service-orchestration.jpeg)

    <Callout type="info">
      **What's happening?** Record mode uses eBPF to intercept HTTP requests *and* Postgres queries at the network level. No SDK or code changes needed in your Go app.
    </Callout>

    Start Keploy in record mode. `-c` sets your run command, `--container-name` tells Keploy which container to watch.

    ```bash
    keploy record -c "docker compose up" --container-name "muxSqlApp" --build-delay 50
    ```
  </CodeTrigger>
</Step>

<Step number={4} title="Generate Test Cases">
  <CodeTrigger file="mock.yaml">
    Just use your API normally. Keploy captures each call as a test case *and* records the Postgres traffic as a mock.

    Open a new terminal and hit the API:

    <CodeGroup>
      <CodeBlock label="Terminal" filename="terminal">
      ```bash
      curl --request POST \
        --url http://localhost:8010/product \
        --header 'content-type: application/json' \
        --data '{
          "name":"Bubbles",
          "price": 123
        }'
      ```
      </CodeBlock>
    </CodeGroup>

    You should receive a response like this:
    ```json
    {
      "id": 1,
      "name": "Bubbles",
      "price": 123
    }
    ```

    <Callout type="tip">
      **One `curl` = one test case + one DB mock.** Compare that to writing a `TestCreateProduct` func, setting up `sqlmock`, and asserting JSON by hand. Keploy did all of that from real traffic.
    </Callout>
  </CodeTrigger>

  ### The Generated Artifacts

  <Note>
    <CodeTrigger file="test-1.yml">
      **1. The Test Case**
      Keploy has now generated a new test file test-1.yml. This file contains the captured request and the expected response, including the exact body and status codes.
    </CodeTrigger>

    **2. The Mocks**
    <CodeTrigger file="mock.yaml">
      Alongside the test case, Keploy also captures external dependencies (like your Postgres database calls) as mocks. These are stored in mock.yaml and allow you to run tests without a live database later.
    </CodeTrigger>
  </Note>

  <CodeTrigger file="test-2.yml">
    ### Fetch Product from Catalog

    Open a new terminal and fetch the products from the catalog:

    ```bash
    curl --request GET --url http://localhost:8010/products
    ```

    Example response:
    ```json
    [{"id": 1, "name": "Bubbles", "price": 123}]
    ```

    Or just type http://localhost:8010/products in your browser. Your choice!

    Spotted the new test and mock files in your project? High five!
  </CodeTrigger>
</Step>

<Step number={5} title="Run the Tests">
  <CodeTrigger file="test-2.yml">
    <Callout type="info">
      **What's happening?** Keploy replays recorded requests and swaps in captured DB mocks. No live Postgres needed. Tests are fully deterministic across local, CI, and teammate machines.
    </Callout>

    Stop the record terminal, then run the test suite. The `--delay` gives Postgres time to initialize.

    ```bash
    keploy test -c "docker compose up" --container-name "muxSqlApp" --delay 10
    ```
  </CodeTrigger>

  <CodeTrigger file="test-2.yml">
    A successful run will output the test report summary in your terminal:

    ![Keploy Success Console](/assets/quickstart/go/docker/mux-sql/keploy-success-console.jpeg)
  </CodeTrigger>
</Step>

## What you just built

<Celebrate>
  You went from zero to a full integration suite without writing a single test function or wiring up mock layers. All from real traffic!
</Celebrate>

<Note>
  **Keploy under the hood:**

  **Intercepted** HTTP + Postgres traffic via eBPF

  **Generated** test cases as version-controllable YAML

  **Mocked** real DB responses for deterministic replay

  **Validated** responses match the original recordings
</Note>

## Troubleshooting

<Alert variant="destructive">
  Getting an EOF or Connection Error? Ensure your `main.go` host is set to `postgres` and that your Docker containers are healthy. Run `docker ps` to verify.
</Alert>
